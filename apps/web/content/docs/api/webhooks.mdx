## Overview

ScreenshotAPI supports two kinds of webhooks:

1. **Customer webhooks** — You register endpoints to receive real-time notifications (e.g., when a screenshot completes).
2. **Payment webhooks** — Internal Polar webhooks that process credit purchases and subscription events.

---

## Customer Webhooks

Register your own HTTP endpoints to receive event notifications from ScreenshotAPI. This is useful for asynchronous workflows where you want to be notified when a screenshot is ready rather than polling for results.

### Register a Webhook Endpoint

```
POST /api/v1/webhooks
```

Creates a new webhook endpoint. Requires session authentication. Returns the endpoint details including a **webhook secret** used to verify incoming deliveries.

#### Request Body

| Field | Type | Required | Description |
|---|---|---|---|
| `url` | `string` | Yes | The HTTPS URL that will receive webhook events |
| `events` | `string[]` | Yes | Event types to subscribe to (e.g., `["screenshot.completed"]`) |

#### Response

```json
{
  "id": "wh_abc123",
  "url": "https://example.com/webhooks/screenshots",
  "events": ["screenshot.completed"],
  "secret": "whsec_MmY3ZjRhNjgtYzUwYi00..."
}
```

<Callout type="warn">
Store the `secret` value securely — it is only returned once when the endpoint is created. You'll need it to verify webhook signatures.
</Callout>

### Supported Events

| Event | Description |
|---|---|
| `screenshot.completed` | Fired when a screenshot has been successfully captured |

### Webhook Payload

When an event fires, ScreenshotAPI sends a `POST` request to your endpoint with a JSON body:

```json
{
  "event": "screenshot.completed",
  "timestamp": "2026-03-24T15:30:00.000Z",
  "data": {
    "screenshotId": "screenshot_xyz",
    "url": "https://example.com",
    "format": "png",
    "width": 1280,
    "height": 720
  }
}
```

### Signature Verification

Every webhook delivery is signed with **HMAC-SHA256** using the secret provided when you created the endpoint. The signature is sent in the `x-webhook-signature` header.

To verify a delivery, compute the HMAC-SHA256 of the raw request body using your webhook secret and compare it to the header value:

<Tabs groupId="sdk" persist items={['JavaScript', 'Python']}>
<Tab value="JavaScript">
```typescript
import { createHmac, timingSafeEqual } from 'node:crypto'

function verifyWebhook(body: string, signature: string, secret: string): boolean {
  const expected = createHmac('sha256', secret).update(body).digest('hex')
  return timingSafeEqual(Buffer.from(signature), Buffer.from(expected))
}
```
</Tab>
<Tab value="Python">
```python
import hmac
import hashlib

def verify_webhook(body: bytes, signature: str, secret: str) -> bool:
    expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
    return hmac.compare_digest(signature, expected)
```
</Tab>
</Tabs>

<Callout type="warn">
Always verify the signature before processing a webhook delivery to prevent spoofed requests.
</Callout>

## Payment Webhooks (Polar)

## Webhook Endpoint

```
POST /api/webhooks/polar
```

This endpoint receives events from Polar. All events are verified using the [Standard Webhooks](https://www.standardwebhooks.com/) signature verification method with the `webhook-id`, `webhook-timestamp`, and `webhook-signature` headers.

## Handled Events

### `checkout.updated`

Fired when a checkout session is updated. We process this event when the checkout `status` becomes `succeeded`.

**Processing logic:**

1. Verify the webhook signature using the Polar webhook secret.
2. Check that the event type is `checkout.updated` and status is `succeeded`.
3. Extract `userId`, `credits`, and `packId` from the checkout metadata.
4. Add the specified number of credits to the user's account.
5. Record a `purchase` transaction in the credit history.

## Security

### Signature Verification

Every incoming webhook request is verified using the Standard Webhooks specification. The following headers are validated:

```
webhook-id: msg_abc123
webhook-timestamp: 1234567890
webhook-signature: v1,abc123def456...
```

If the signature is invalid, the webhook returns a `403` error and the event is discarded. This prevents replay attacks and ensures events genuinely originate from Polar.

### Error Responses

| Status | Body | Description |
|---|---|---|
| `403` | `{"error": "Invalid webhook signature"}` | Signature verification failed |

### Success Response

```json
{
  "received": true
}
```

## Webhook Flow

Here's the complete payment flow involving webhooks:

1. User selects a credit pack in the dashboard.
2. Frontend calls `POST /api/v1/credits/purchase` with the `packId`.
3. Server creates a Polar Checkout session with user metadata.
4. User is redirected to Polar Checkout to complete payment.
5. After successful payment, Polar sends a `checkout.updated` event to our webhook.
6. Webhook handler verifies the signature, extracts the metadata, and adds credits.
7. User is redirected back to the dashboard where their updated balance is visible.

## Testing Webhooks Locally

If you're developing against the ScreenshotAPI platform or running it locally, you can use a tunneling service like ngrok to forward webhook events:

```bash
ngrok http 3001
```

Then register the ngrok URL as your webhook endpoint in the Polar dashboard (e.g., `https://your-id.ngrok.io/api/webhooks/polar`). Set the `POLAR_WEBHOOK_SECRET` environment variable to match the secret configured in Polar.

## Idempotency

Webhook handlers are designed to be idempotent. If the same event is delivered multiple times (which Polar may do for reliability), credits are only added once because each transaction references the unique Polar checkout ID.
