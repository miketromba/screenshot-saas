## Overview

ScreenshotAPI uses Stripe webhooks to process payment events. When a user purchases credits through Stripe Checkout, a webhook event is sent to our server to fulfill the order by adding credits to the user's account.

<Callout type="info">
Webhooks are handled server-side and are not something you need to configure as an API consumer. This documentation is provided for transparency and for developers integrating with the ScreenshotAPI platform.
</Callout>

## Webhook Endpoint

```
POST /api/v1/webhooks/stripe
```

This endpoint receives events from Stripe. All events are verified using the `stripe-signature` header and the webhook signing secret configured on the server.

## Handled Events

### `checkout.session.completed`

Fired when a user completes a credit pack purchase through Stripe Checkout.

**Processing logic:**

1. Extract `userId`, `credits`, and `packId` from the session metadata.
2. Add the specified number of credits to the user's account.
3. Record a `purchase` transaction in the credit history.

### `payment_intent.succeeded`

Fired when a payment intent succeeds. Used specifically for auto top-up payments.

**Processing logic:**

1. Check if the payment intent metadata contains `type: "auto_topup"`.
2. If so, extract `userId` and `credits` from the metadata.
3. Add the credits to the user's account.
4. Record an `auto_topup` transaction in the credit history.

## Security

### Signature Verification

Every incoming webhook request is verified using Stripe's signature verification:

```
stripe-signature: t=1234567890,v1=abc123def456...
```

If the signature is missing or invalid, the webhook returns a `400` error and the event is discarded. This prevents replay attacks and ensures events genuinely originate from Stripe.

### Error Responses

| Status | Body | Description |
|---|---|---|
| `400` | `{"error": "Missing stripe-signature header"}` | No signature header was provided |
| `400` | `{"error": "Invalid webhook signature"}` | Signature verification failed |

### Success Response

```json
{
  "received": true
}
```

## Webhook Flow

Here's the complete payment flow involving webhooks:

1. User selects a credit pack in the dashboard.
2. Frontend calls `POST /api/v1/credits/purchase` with the `packId`.
3. Server creates a Stripe Checkout session with user metadata.
4. User is redirected to Stripe Checkout to complete payment.
5. After successful payment, Stripe sends a `checkout.session.completed` event to our webhook.
6. Webhook handler verifies the signature, extracts the metadata, and adds credits.
7. User is redirected back to the dashboard where their updated balance is visible.

## Auto Top-Up Flow

1. User configures auto top-up with a threshold and credit pack in the dashboard.
2. When a screenshot is taken and the credit balance drops below the threshold, the server creates a Stripe PaymentIntent with auto-topup metadata.
3. Stripe processes the payment against the user's saved payment method.
4. On success, Stripe sends a `payment_intent.succeeded` event.
5. Webhook handler verifies the event and adds credits to the user's account.

## Testing Webhooks Locally

If you're developing against the ScreenshotAPI platform or running it locally, you can use the Stripe CLI to forward webhook events:

```bash
stripe listen --forward-to localhost:3001/api/v1/webhooks/stripe
```

The CLI will output a webhook signing secret (starting with `whsec_`) that you should set as your `STRIPE_WEBHOOK_SECRET` environment variable.

## Idempotency

Webhook handlers are designed to be idempotent. If the same event is delivered multiple times (which Stripe may do for reliability), credits are only added once because each transaction references the unique Stripe session or payment intent ID.
